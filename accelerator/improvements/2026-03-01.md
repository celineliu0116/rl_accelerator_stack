# Improvement Log 2026-03-01

## Scope

This log records the completion of full DMA integration for `MatmulAcceleratorSystolic`.
The previous systolic path had a placeholder feed that tied `a_left` and `b_top` to zero.
Today's work replaces that with a working DMA-fed skew controller that actually drives
the 4×4 PE array from memory, producing correct inference results.

## Problem Statement

The systolic accelerator existed structurally but was not computing.
The placeholder `MatmulAcceleratorSystolic.sv` had:

1. `a_left` and `b_top` hardwired to zero
2. DMA signals `dma_addr` and `dma_re` tied off
3. A timer-based fake busy signal that just waited a fixed number of cycles

This meant the systolic path was passing inference only because results were being
produced by a residual path, not by actual PE accumulation.
The 2026-02-24 numbers for the systolic path were therefore not measuring real
systolic computation.

## Implementation

### DMA prefetch scheduler

Added a circular prefetch buffer with a 7-state FSM.

```text
S_IDLE -> S_POP -> S_LATCH -> S_PREFILL -> S_COMPUTE -> S_DRAIN -> S_DONE
```

- `S_POP` and `S_LATCH` drain the command FIFO and latch the active command fields
- `S_PREFILL` issues DMA reads to fill the prefetch buffer before compute begins
- `S_COMPUTE` iterates over K steps, unpacking INT8 values and feeding the array
- `S_DRAIN` runs for N-1 extra cycles to flush the systolic pipeline
- `S_DONE` holds results and signals software

A tag queue maps in-flight DMA responses to prefetch buffer slots.
Buffer slots are marked valid on response and consumed in order during compute.

### Skew controller

The systolic array requires data to enter diagonally.
Row `r` of `a_left` receives its data `r` cycles after row 0.
Column `c` of `b_top` receives its data `c` cycles after column 0.

Implemented with shift registers:

```text
a_left[0] = a_in[0]            (no delay)
a_left[1] = a_sr[1][0]         (1 cycle delay)
a_left[2] = a_sr[2][1]         (2 cycle delay)
a_left[3] = a_sr[3][2]         (3 cycle delay)
```

Same pattern for `b_top`.
This ensures PE(r, c) receives matching k-index data from both directions.

### Data unpacking

DMA returns 256-bit beats.
Each beat delivers 8 packed INT8 words.
The low 128 bits carry X data (4 batch rows × 4 k-steps).
The high 128 bits carry W data (4 neurons × 4 k-steps).

Each byte is sign-extended to 16 bits before feeding the PE inputs,
matching the `DATA_W=16` interface of `SystolicPE`.

### Clear accumulator

`clear_acc` is asserted for exactly one cycle at the very first sub-step
of the very first DMA beat of each new command, resetting PE accumulators
before the new tile begins accumulating.

### Performance counters

Added per-command logging:

```text
ACCEL_PERF_SYSTOLIC cmd=N k_limit=K busy=B compute=C stall=S dma_req=D
```

- `busy` is cumulative wall-clock cycles since simulation start where the accelerator was active
- `compute` is MAC-active cycles for that command
- `stall` is cycles where compute was ready but no buffer slot was valid

## Key design decisions

1. **Interleaved DMA format**: one 256-bit beat delivers both X and W slices,
   matching the data layout produced by `pack_input_batch` and `pack_weight_block`
   in `inference_bare.c`.

2. **Verilator compatibility**: all temporaries in `always @(posedge clk)` blocks
   are declared as module-level `reg`, not `automatic`. Verilator does not support
   `automatic` variables inside clocked blocks.

3. **Drain length**: N-1 = 3 extra cycles after the last compute step to flush
   the systolic pipeline before latching `c_out`.
