# Improvement Log - March 2, 2026

## Overview
This log documents a major breakthrough in system stability and the transition to a fully decoupled "Train on Silicon" workflow. A critical hardware bug in the CPU pipeline was identified and resolved, enabling robust -O3 execution for the first time.

## 1. Dual-Issue CPU Pipeline Resolution (The "Slip-Stall" Bug)
### The Problem
During `-O3` simulation of the generic runtime, the system would consistently hang or report zero accuracy. Tracing the instruction retirement showed that the CPU was entering infinite loops in code that worked perfectly at `-O0`.

The root cause was identified in the `slip_stall` logic within `Riscv151.v`. In a dual-issue design, if the front-end cannot issue a 64-bit instruction bundle in a single cycle (due to hazards or lane-eligibility), it must "slip" the second instruction to the next cycle using a `half_fetched` state.

**The Bug:** The pipeline was incorrectly failing to trigger a `slip_stall` when the first instruction in a bundle was a **Branch** predicted **NOT TAKEN**. Because the branch was predicted not-taken, the next instruction in the bundle was technically on the "right path," but the logic wasn't saving it for the next cycle. This caused the CPU to effectively "skip" the instruction following such branches (often crucial loop increments or data loads).

### The Fix
The `slip_stall` assignment was modified to correctly recognize that branches predicted as not-taken REQUIRE a slip stall if the lane1 instruction cannot be issued immediately.

**Final RTL Logic:**
```verilog
assign slip_stall = have_bundle && !half_fetched && !flush_id &&
                    !freeze_ext && !lane1_issue_en &&
                    !((is_branch_id | is_j_id) & pred_id);
```
By adding `& pred_id`, we ensure that:
*   If a branch is predicted **Taken**, we do **not** slip (lane1 is on the wrong path).
*   If a branch is predicted **Not-Taken**, we **DO** slip (lane1 is on the right path and must be executed).

## 2. Decoupled "Train on Silicon" Workflow
We have successfully moved away from "Hardcoded Firmware." The system now supports a generalized model execution flow.

*   **Model Descriptor Blob:** Introduced a binary model format (`model_format.h`) that includes self-describing headers (layers, dimensions, activations).
*   **Generic Runtime:** `inference_generic.c` now parses the model blob at runtime. It orchestrates DMA transfers and configures accelerator MMIO registers dynamically based on the model geometry.
*   **Python Exporter API:** Developed `export_model.py` which allows researchers to train a model in PyTorch/NumPy and "flash" the model to the SoC without touching a single line of C code or Verilog.

## 3. Systolic Array Verification & Signoff
The **Weight-Stationary Systolic Array** implementation has reached production-ready status.

*   **Performance:** Verified `stall=0` compute cycles across standard matrix operations. The DMA-fed skew controller now provides a perfect data stream to the 4x4 array.
*   **Physical Signoff:** Completed OpenLane PAR flow for the full SoC.
    *   **Target:** 100MHz (10ns)
    *   **Verified Signoff:** ~60MHz (17ns) on Sky130 process (due to high routing density).
    *   **Utilization:** Achieved **~1.0 IPC** on the dual-issue core during optimized matrix compute phases.
